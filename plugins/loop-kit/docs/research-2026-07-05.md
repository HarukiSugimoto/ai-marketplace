# Loop Engineering 調査レポート(2026-07-05)

deep-research ワークフローによる網羅調査の結果。111エージェント・28ソース取得・
140クレーム抽出・上位25件を3票の反証検証にかけ、**25件全てが3-0で確認、反証ゼロ**。
以下の知見が loop-kit の設計根拠。

## 要旨

**Loop engineering(ループエンジニアリング)** とは、コーディングエージェントに
単発のプロンプトを与えるのではなく、**エージェントを繰り返し駆動する「ループ」自体を
設計する**実践のこと。実践は3系統に整理できる:

1. **外側シェルループ** — headless `claude -p` を bash while/for で回す(Ralph 古典形)
2. **セッション内 Stop hook ループ** — セッション終了をブロックして同一プロンプトを再投入
3. **検証ループ** — pass/fail チェック(テスト等)を与えて収束させる。Anthropic 公式の
   agent loop(gather context → take action → verify work → repeat)の思想的中核

## 検証済みの知見(すべて high confidence, 3-0)

### 起源

1. **用語の普及**: Peter Steinberger (@steipete) の 2026-06-07 の X 投稿
   「You should be designing loops that prompt your agents」(24時間で約500万ビュー)が契機。
   ただし同週に Anthropic の Boris Cherny がほぼ同旨の発言をしており単独発案ではない。
   なお公式 docs は「loop engineering」という語を使っていない(コミュニティ側の呼称)。
2. **技法の原型**: Geoffrey Huntley の **Ralph Wiggum loop**(ghuntley.com/ralph、2025-07-14)。
   純粋形は `while :; do cat PROMPT.md | claude-code ; done`。名前はシンプソンズ由来。
   核心の性質は「deterministically bad in an undeterministic world」= 失敗が予測可能なので
   プロンプト調整で直せる(Failures Are Data)。
3. **Huntley 自身の限定**: 「There's no way in heck would I use Ralph in an existing code base」
   — Ralph はグリーンフィールド向け(2025年7月時点の本人の立場。その後コミュニティは
   ブラウンフィールド適用も報告)。

### 実装(公式)

4. **Anthropic 公式 ralph-wiggum プラグイン**: 外側 bash ループではなく **Stop hook** で実装。
   hook がセッション終了をブロック(JSON で `decision: block`)し、同一プロンプトを reason 経由で
   再投入。反復間の記憶はファイルと git 履歴のみ。`/ralph-loop "<prompt>" --max-iterations N
   --completion-promise "TEXT"` + `/cancel-ralph`。状態は `.claude/ralph-loop.local.md`。
   **completion-promise は完全一致マッチのため複数終了条件に使えず、max-iterations を
   第一の安全機構とせよ**と README が規定。
5. **適否の公式ガイド**: 向く=成功条件が明確/自動検証がある/反復で良くなる/放置できる
   グリーンフィールド。向かない=人間の判断が必要/成功条件不明瞭/一発勝負/本番デバッグ。
   哲学3点: Iteration > Perfection、Failures Are Data、Operator Skill Matters。
6. **headless CLI の4要素**: `claude -p`(非対話)/ `--continue`・`--resume <session_id>` /
   `--output-format json`・`--json-schema`(機械可読出力)/ `--allowedTools`・
   `--permission-mode`(無人実行)。公式例:
   `claude -p "Run the test suite and fix any failures" --allowedTools "Bash,Read,Edit"`。
   注意: docs はスクリプト用途に `--bare` を推奨し始めており CLI 仕様は変わりうる。
7. **ファンアウトの公式推奨**: `for file in $(cat files.txt); do claude -p "Migrate $file ...
   Return OK or FAIL." --allowedTools "Edit,Bash(git commit *)"; done`。
   **まず2〜3ファイルで試してプロンプトを改善してから全体適用**という段階手順も公式。
8. **検証ループ = 自律性の分水嶺**: 「pass/fail を返すチェックを与えれば、人間が検証ループで
   ある状態を置き換えられる」(公式 best-practices)。評価手段はルールベース/ビジュアル/
   LLM-as-judge の3種(judge は公式も「頑健でなくコスト重い」と留保)。
9. **ネイティブなループ制御2種**: (1) Stop hook — 8回連続ブロックで強制終了する組み込みの
   無限ループ防止付き(`CLAUDE_CODE_STOP_HOOK_BLOCK_CAP` で変更可)。(2) `/goal` — 別の
   評価モデル(既定 Haiku)が毎ターン後に条件を yes/no 判定して継続を制御。
10. **終了条件設計(サードパーティの解法)**: frankbria/ralph-claude-code は
    「自然言語の完了指標 >= 2 AND Claude 自身の EXIT_SIGNAL: true」の**二重ゲート**で終了。
    加えてレート制限・サーキットブレーカー・トークン予算の多層セーフガード。
    → loop-kit の run-loop.sh は「完了宣言 AND 検証コマンドパス」の二重ゲートとして採用。

## 留意点(caveats)

- **日本語圏ソース**: Zenn/Qiita の記事は取得したが、3票検証を通過したクレームは
  すべて英語圏の一次ソース由来。下記の日本語記事は「参考(未検証)」扱い
- **時間感度**: この分野は動きが速い。Huntley の brownfield 否定は2025年7月時点。
  CLI 仕様(`--bare` 等)も変わりうる
- **効果の実証性**: 「$50k 契約を $297 で完了」等の逸話は未検証。自己修正の有効性一般にも
  学術的異論あり(Huang et al., ICLR 2024)
